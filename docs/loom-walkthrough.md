# Loom Walkthrough Script (~4 minutes)

---

## 0:00 – 0:30 | What the system does

Hi, I'm going to walk you through the Scaler AI Persona system.

This is a live voice agent and chat interface that answers questions about my background and public GitHub projects. It also checks real calendar availability and books slots — all without using a single LLM API call.

Every answer is grounded in a pre-built corpus. There's no generation, no hallucination risk, and zero per-query cost.

---

## 0:30 – 1:15 | Architecture overview

Here's the architecture, top to bottom.

On the left is the chat interface — a Next.js page at `/chat` that sends POST requests to `/api/chat`.

On the right is the voice entry point — a Twilio phone number that hits `/api/voice` when a call comes in. Each spoken turn is a separate HTTP POST to `/api/voice/response`.

Both paths feed into an orchestration layer. The chat orchestrator is simple: retrieve and reply. The voice orchestrator is a state machine — it moves through greeting, Q&A, booking date, booking time, and done phases.

Below that are three tools: retrieval, availability, and booking. These are independently testable and share no state.

At the bottom is the knowledge layer: a BM25 plus n-gram search over a 25-chunk corpus, a Google Calendar client with a mock fallback, and an in-memory session store for active calls.

The ingestion pipeline runs offline — `npm run ingest` reads my resume and GitHub repos and writes the corpus.

---

## 1:15 – 2:00 | One hard problem: no-LLM grounding

The hardest problem I solved was building a natural-sounding voice agent without an LLM.

Most people reach for GPT or Claude for this — but that introduces latency, cost, and hallucination risk. I wanted a system that was provably grounded with zero per-query cost.

The approach: pure BM25 retrieval hybridised with bigram Jaccard similarity. BM25 handles term-frequency relevance; bigram overlap catches phrase-level matches that BM25 misses.

The voice orchestrator retrieves the top corpus chunk, truncates it to under 400 characters, and speaks it verbatim using Amazon Polly's Joanna voice through Twilio. There's no rephrasing — what you hear is exactly what the corpus contains.

This means the system cannot hallucinate. There is no model to jailbreak. And the prompt injection guard — 13 regex patterns checked before every search — means even adversarial queries never touch the retrieval layer.

## 3:00 – 3:45 | Voice agent and booking

The voice agent follows the same retrieval logic but through a state machine.

When a call comes in, Twilio sends a POST to `/api/voice`. The server responds with TwiML — a welcome message wrapped in a `<Gather>` tag that listens for speech.

The user says: "Tell me about Aforro Dashboard." That transcript goes to `/api/voice/response`. The voice orchestrator runs the same `retrieve()` function and speaks the result.

If the user says "book a call," the state machine transitions from Q&A mode to booking_date. The server parses the date from natural speech — "tomorrow," "next Tuesday" — then checks availability via the Google Calendar freebusy API with a 10-minute in-memory cache. If credentials aren't configured, it falls back to synthetic 9-to-5 slots so the full flow works in dev.

The user picks a time. The server parses "2pm" or "10:30am," creates a Google Calendar event through the JWT-authenticated client, and returns a confirmation code.
